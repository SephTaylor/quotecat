# v1.2.7 — Barcode Pricebook Scanner + Shareable Health Check Card

**Status:** Planned 2026-06-01. Carries forward from the original v1.2.6 scope — scanner was moved out of v1.2.6 to keep that release laser-focused on Pricing Health Check. All code exploration findings from the v1.2.6 planning round are preserved below — no work is lost.

**Theme:** Two features that, paired with v1.2.6's Pricing Health Check, complete the financial-intelligence platform from end to end. The scanner captures the contractor's *present* (real purchase prices at the store). The shareable card turns the audit into *organic reach* (a Strava-style moment of recognition that travels).

---

## The strategic case

After v1.2.6 ships, QuoteCat does this:

- **Free tier:** real per-quote margin + estimated profit + free tool calculators
- **Pro tier:** all of the above + Pricing Health Check audits the contractor's past quotes

v1.2.7 adds two pieces that complete the loop:

- **Barcode Pricebook Scanner** captures the contractor's *real purchase prices at the point of purchase* via the phone camera. Mobile-first by definition; desktop competitors can't match it. Over months, the pricebook becomes a personal pricing intelligence database — the prices the contractor actually pays, not generic catalog data.

- **Shareable Health Check summary card** turns the private audit moment into a public one. Strava-style: *"7 of my last 23 quotes were underpriced — $4,800 on the table. — calculated by QuoteCat"*. No client names. Each share is organic growth and a moment of credibility for the brand. Pairs with the calculator share infrastructure already shipped on quotecat.ai.

## Verified findings from code exploration (preserved from v1.2.6 planning)

These were verified by reading actual code during the v1.2.6 planning round. Re-verify lightly when starting v1.2.7 work in case anything has moved.

### Pricebook items schema (`lib/types.ts:245+`)

`PricebookItem` includes `sku?: string`. Confirmed — that's where scanned barcodes get stored.

### SKU lookup gap (`lib/pricebook.ts`)

`searchPricebookItems()` does **substring LIKE matching** on name/description/sku — wrong for barcodes because scanning "123" would match "SKU-1234-XYZ". Must add `getPricebookItemBySkuExact(sku: string): Promise<PricebookItem | null>` that runs `SELECT * FROM pricebook_items WHERE sku = ? AND deleted_at IS NULL LIMIT 1`. Trim the input.

### Pricebook screen structure (`app/(main)/price-book.tsx`)

Create/edit uses a **modal**, not a separate screen (lines 159-224 for handlers, line 450 for modal). The scanner integrates by feeding the existing modal:

- Match found → open existing edit modal pre-populated (reuse `handleEditItem` near line 179)
- No match → open create modal with `sku` field pre-filled (reuse `handleAddItem` near line 176)

UI placement recommendation: barcode-scan icon in the **header right**, next to existing nav controls. Not a FAB (conflicts with sheet UI). Tier check: if Free, `presentPaywallAndSync()`; if Pro, open scanner.

### Cloud sync (`lib/pricebook.ts:87-111`)

`savePricebookItem()` already auto-syncs Pro user items to Supabase via `uploadPricebookItem()`. Barcode-created items follow this path with zero extra work.

### Camera infrastructure

- **`expo-camera` is NOT installed** in `package.json`. Need `npx expo install expo-camera`. SDK is 54.0.19.
- iOS `NSCameraUsageDescription` already present at `app.json:62` but scoped to logo photos. Broaden to: *"QuoteCat uses the camera to take photos of your company logo and to scan product barcodes for your pricebook."*
- ML Kit dependency that Google Play flagged in v1.2.4 (deprecated edge-to-edge APIs) comes transitively from `expo-camera`. Installing it makes the ML Kit dependency intentional → resolves the deprecation flag cleanly without removing `expo-dev-client`.

### Barcode types worth supporting

`CameraView` with `barcodeScannerSettings={{ barcodeTypes: ['upc_a','upc_e','ean13','ean8','code128','code39'] }}`. Covers North American retail (UPC), global retail (EAN), and warehouse/supplier tags (Code-128, Code-39). The scanner is geography-agnostic — works in every country because it doesn't depend on a product database.

## Also shipping — analytics user identity + missing instrumentation

Small infrastructure changes made during the v1.2.7 work cycle (2026-06-05). Not user-facing; sets up better data for every dashboard and crash report afterward.

### What changed

| File | Change |
|---|---|
| `lib/app-analytics.ts` | Added `identifyUser(userId, { email, tier })` and `resetAnalyticsUser()` helpers — drive **both** PostHog and Sentry from one call |
| `app/(auth)/sign-in.tsx` | Calls `identifyUser` after tier resolution, both OAuth and email/password paths |
| `app/(auth)/sign-up.tsx` | Calls `identifyUser` after signup, both Google and email/password paths |
| `lib/user.ts` | Calls `resetAnalyticsUser` in `signOutUser` so events from the next account don't merge with the previous user |
| `app/(forms)/quote/[id]/review.tsx` | `trackEvent(AnalyticsEvents.REVIEW_OPENED)` fires on mount (was declared in the catalog but never wired — left the Quote Lifecycle funnel's third step empty) |

### Why

Pre-fix: PostHog and Sentry both ship events anonymously — `distinct_id` is a per-install fingerprint, not the Supabase user ID. Two consequences:

1. Dashboards built on 2026-06-05 (App Health, Quote Lifecycle Funnel) attribute every event to an anonymous device ID — can't filter by tier or look at single-user journeys.
2. Sentry crash reports don't show which account hit them — once real users start hitting bugs, this matters for triage.

Post-fix: every subsequent PostHog event ties to user ID + email + tier, and every Sentry exception carries `{ id, email, tier }`. Sign-out cleanly resets so account-switching during testing doesn't cross-contaminate.

### Verification before submit

1. Sign in with email/password → PostHog **Activity** tab → confirm `app_opened` `distinct_id` is the user's Supabase UUID, not a random string
2. Sign out → sign in as a **different** account → confirm events show the new user, not the old one
3. Trigger a caught error path (force an offline sync or malformed PDF) → Sentry event should show `user.id` + `user.email` populated
4. Open a quote → tap into the review screen → PostHog should receive a `review_opened` event with `quoteId` property

### Effort

~30 minutes already done. Verification ~15 min. No additional dev time pulled from scanner/card scope.

## Also shipping — Stripe Connect collection for all tiers (decided 2026-06-05)

Currently Stripe Connect onboarding is exclusively `portal.quotecat.ai/dashboard/settings`, and the portal is Premium-only (`dashboard/layout.tsx:50` checks `profile.tier === 'premium'`). Net effect today: Free and Pro contractors **cannot accept card payments**. Their clients on the public `/pay/[id]` page see Zelle/Venmo/cash/check only.

This release ungates Stripe Connect setup so any tier can configure it from the mobile app.

### The strategic case

1. **Payment acceptance is table-stakes.** Square, HoneyBook, Wave, and every modern quoting/invoicing tool let any user accept card payments. Gating it to Premium reads as stingy and creates a "forced upgrade" friction that breeds resentment, not loyalty.
2. **The "no QuoteCat fees, ever" hero claim is more honest when *anyone* can collect cards.** Saying "we don't take a cut + Pro/Premium only can collect cards" doesn't pair well.
3. **Premium's actual moat isn't payment processing** — it's team management, Drew quote-building, contracts, the portal dashboard, QuickBooks, the financial-intelligence layer. Premium can lose Stripe Connect exclusivity without weakening positioning.
4. **It removes a friction point in the Free → paying contractor funnel.** Free contractors who lose a client because they couldn't accept cards either churn or grudgingly upgrade. Both are bad.

### Decision matrix (locked)

| Tier | Stripe collection | Share as Link | Payment client experience |
|---|---|---|---|
| **Free** | ✓ (configure in mobile app) | ✗ — stays Pro+ | PDF includes a payment QR code + short URL to `/pay/[id]` |
| **Pro** | ✓ | ✓ — clickable web link | Client taps text/email link → Stripe checkout |
| **Premium** | ✓ (mobile OR portal) | ✓ | Same as Pro + manage Stripe from web |

**Why "Share as Link" stays Pro+:** It's a real, concrete Free→Pro upgrade hook that doesn't depend on artificial gating. The pitch becomes *"Free contractors print a PDF with a payment QR. Pro contractors send a text link the client taps to pay. Same Stripe rates, way less friction."* Friction-removal sells better than feature-counting.

### Apple compliance

Stripe Connect onboarding in the mobile app is **not** Apple IAP territory. IAP rules govern QuoteCat charging the user for in-app digital goods. Stripe Connect is the contractor setting up their own merchant account to receive payments from their own clients — a business operation. Square, HoneyBook, Wave, Stripe Dashboard, and many others do this on iOS without issue. No App Store risk.

## Day 2 hard cut criterion (preserved from v1.2.6 plan)

At the end of the scanner build day, run a concrete reliability test on a real iPhone (release build, not Metro dev):

> Scan **5 consecutive barcodes** of varied types — at minimum: UPC-A on a paint can, EAN-13 on a screw box, Code-128 on a hardware/supplier tag, and 2 more from common products around the house. **At least 4 of the 5 must decode correctly within ~3 seconds each** with the camera pointed at the barcode at a reasonable angle.

If the scanner misses that bar in v1.2.7 → ship just the Shareable Card and slip the scanner further. (In a single-feature-style release this is less catastrophic than it was in v1.2.6 because we always had a backup feature in flight.)

### Android smoke test (~2 hours, REQUIRED before submission)

ML Kit's Android path differs enough from iOS AVFoundation that a zero-test Android submission is avoidable risk. Test on a physical Samsung Galaxy or equivalent. Same 4-of-5 success criterion as the iOS cut test.

## Resolved decisions (preserved)

1. **Barcode tier gating:** Pro-only. Free users tap → paywall. Matches the Pro = automation pattern.
2. **UI placement:** Header right icon on Price Book screen. Doesn't disrupt existing "+ New" button.
3. **Permission scope:** Broaden the existing logo permission rather than add a second one — Apple reviewers prefer fewer, broader strings.
4. **No product database in v1.** Contractor is at the store holding the product — they're the source of truth. UPC databases are US-centric anyway. Scan-and-remember works in every country with zero localization.
