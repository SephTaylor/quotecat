# v1.2.6 — Pro Feature Release Plan

**Status:** Drafted 2026-06-01. Build starts after v1.2.5 promotes to production.

**Theme:** Two new Pro features that compound the financial-intelligence positioning — one that surfaces the past (Pricing Health Check) and one that captures the present (Barcode Pricebook Scanner).

---

## The strategic case (the moat)

The two features together do something competitors structurally can't match.

**The product moat (already in place):**
- Real per-quote margin display on every quote (Free in v1.2.5)
- Overhead-loaded labor rate built into pricing math
- A custom pricebook that travels with the contractor

**What v1.2.6 adds:**

- **Pricing Health Check** turns the contractor's *historical quote data* into actionable insight. "You sent 11 quotes last month — 3 of them were underpriced by an estimated $2,400 combined." Nobody else has this view. Jobber and Housecall Pro show revenue trends, not margin diagnostics.

- **Barcode Pricebook Scanner** captures the contractor's *real purchase prices at the point of purchase*. A desktop-first competitor cannot do this. Over months, this turns the pricebook into a personal pricing intelligence database — the actual prices the contractor pays, not generic catalog data.

Combined: the app gets smarter the longer the contractor uses it. Every job they price adds to the historical analysis. Every store run adds to the pricebook. That compounding accuracy is the moat — and it's mobile-first by definition.

# Feature 1: Pricing Health Check

## What the user sees

A Pro-only screen reachable from the Toolbox under Pro Tools. One tap → analyzes recent quotes → shows ones that were likely underpriced with estimated lost profit. Headline number is the hook:

> *Last 90 days: you sent 11 quotes. 3 likely underpriced. **~$2,420 estimated profit on the table.***

Below: sortable list of flagged quotes with project name, client, total, actual margin vs target margin, and estimated lost profit. Tap to open the quote.

## Analysis rules (v1)

For each quote with `status IN ('sent', 'approved', 'completed')` from the last 90 days:

1. Compute **actual margin** from quote data:
   - `cost = materialEstimate + (labor × laborCostRate / laborRate)`
   - `margin% = (total - cost) / total × 100`
2. Get user's **target margin** from `preferences.overhead.targetProfitMarginPercent` (default 20% if unset)
3. If `actualMargin < targetMargin - 5` → **flag as underpriced**
4. **Estimated lost profit** = `(targetMargin - actualMargin) / 100 × quote.total`
5. Sort by lost profit descending. Show top 10.

Pure function, deterministic, runs locally against SQLite data via `listQuotes()`.

**Edge cases:**
- No target margin set → use 20%, show banner: "Set your target margin for more accurate results"
- No labor cost rate set → assume cost = 70% of labor billed
- Custom line items (no productId) → still computable from `unitPrice × qty`
- Drafts excluded — only quotes that went to a client

# Feature 2: Barcode Pricebook Scanner

## What the user sees

A "Scan barcode" button on the Price Book screen (Pro-only). Tap → camera opens → scan a UPC/EAN barcode at the store. App matches against existing pricebook items by SKU:

- **Match found:** opens that item for editing. "Last saved at $45 — what's it cost today?"
- **No match:** opens new-item form with the barcode pre-filled into SKU. Contractor types name + price + saves.

Next time they scan the same product, the app recognizes it. Over time the pricebook becomes a personal database of "what I actually pay for stuff."

## Design notes

- **No product database lookup in v1.** Contractor is at the store holding the product — they're the source of truth. Avoids the messy "barcode → which Home Depot product" problem entirely.
- **The SKU field already exists** in the `pricebook_items` schema. We're just routing barcode scan results into it.
- **Permission copy** (`NSCameraUsageDescription`): "QuoteCat uses the camera to scan barcodes so you can add products to your pricebook quickly while shopping for materials."

## Library choice

- **Use:** `expo-camera` with barcode detection (the modern Expo barcode API)
- **Don't use:** `expo-barcode-scanner` — deprecated
- **Already pulled in transitively:** The ML Kit dependency Google Play flagged comes from `expo-dev-client`. Adding `expo-camera` makes that dependency actually used — turns a Play Store warning into a legitimate feature.

## Combined release strategy

### Sequencing within v1.2.6

1. **Day 1 (4-5 hrs):** Pricing Health Check — `lib/pricingHealth.ts` + tests, then the screen. Lower risk, doesn't touch native code.
2. **Day 2 (4-5 hrs):** Barcode Scanner — `expo-camera` install, scanner screen, price-book integration. Touches native (requires a rebuild via `expo prebuild` or just a fresh EAS build).
3. **Day 3 (2-3 hrs):** Final testing + commit + EAS build + submit.

**Total: 2-3 working days.**

### What gets cut if scope blows up

In order:
1. Barcode scanner "smart edit" prompt ("last saved at $X — what's it cost today?") — nice but not required for MVP. Can be a v1.2.7 polish.
2. Pricing Health Check dashboard widget (the "3 quotes need review" badge on dashboard). Toolbox entry alone covers the use case.
3. Unit tests for `pricingHealth.ts`. Risky to skip but tolerable if the rule logic is simple enough to verify manually.

Keep both core screens functional. If both blow up, ship just one and slip the other to v1.2.7 — don't ship a half-broken pair.

### Dependencies between the two

None. They can ship together or apart. Pricing Health Check is JS-only; Barcode Scanner needs a native rebuild because `expo-camera` is added. The native rebuild adds a few minutes to the EAS build but no real risk.

## Open decisions before build starts

1. **Pricing Health Check time window.** 90 days, 30 days, or both with a toggle? Start with 90 fixed?
2. **Pricing Health Check threshold.** `actualMargin < targetMargin - 5` proposed. Tighter (3%) catches more quotes. Looser (10%) feels too forgiving.
3. **Pricing Health Check fallback for labor cost.** If user hasn't set `defaultLaborCostRate`, assume 70% of labor billed (proposed), or skip the labor cost factor entirely?
4. **Barcode scanner Pro-or-Free.** Recommended Pro (matches Pro = automation pattern). Could argue Free since it's a small productivity tool.
5. **Barcode scanner entry point.** Just the Price Book screen, or also from inside the quote editor ("scan to add as a line item")? Latter is more powerful but doubles the scope.
6. **Native rebuild timing.** Doing both features at once means one EAS build cycle. If we want barcode scanner shipped first as a v1.2.6.1 hotfix-style release, we'd need two build cycles. Recommend: one combined v1.2.6.
